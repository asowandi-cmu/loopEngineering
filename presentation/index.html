A Note on Vector Matrices and Agent Focus

Why persona framing actually works — it's not metaphorical.

This section gives the mechanistic justification for role prompts.

Role Prompting Research

  • "Role-Play Prompting" (Kong et al., NAACL 2024) — AQuA accuracy: 53.5% → 63.8%
    Assigning an expert persona before zero-shot reasoning tasks measurably improves accuracy on college-level algebra without any examples.
  • "ExpertPrompting" (Xu et al., 2023) — ExpertLLaMA at 96% of ChatGPT
    Generating detailed expert identity descriptions as prompts; fine-tuned open-source model achieves near-parity with ChatGPT (GPT-4 judged).
  • "In-Context Impersonation" (Salewski et al., NeurIPS 2023 Spotlight)
    Domain-expert personas outperform non-experts on matched tasks; also reveals social biases embedded in model representations.
  • "Representation Engineering" (Zou et al., 2023) — 36+ behaviors controlled via vectors
    Linear "control vectors" in residual streams steer honesty, power-seeking, harmlessness — no fine-tuning needed, just representation-level intervention.
The headline is that prompting as an expert changes measurable task performance. Papers 1-3 show prompting-level gains; paper 4 proves the mechanism exists at the representation level.

Mechanism: the Causal Chain

Terminology Choice → Token Embedding → Attention Circuits → Residual Stream → Sparse Feature Activation → Output Distribution

Key insight: "security engineer" literally activates different neural features than "software developer".

Make this concrete: words select trajectories through the model, not just vibes.

Anthropic's Proof

  • "Scaling Monosemanticity" (Anthropic, May 2024) — found features for "code with security vulnerabilities" vs "abstract discussion of security"
    Used sparse autoencoders on Claude 3 Sonnet to extract millions of interpretable features. The model has distinct neural features for recognizing actual code vulnerabilities vs. conceptual security discussions — proving role-context creates genuinely different internal representations.
  • "Refusal in Language Models Is Mediated by a Single Direction" (Arditi et al., 2024) — complex behaviors live in 1D subspaces
    Refusal behavior across 13 open-source models (up to 72B params) is controlled by a single direction in activation space. Erase it → model stops refusing. Add it → model refuses everything. Shows safety behaviors are surprisingly simple and brittle linear structures.
The important point is not interpretability trivia; it is that behavior steering is real and directional. Monosemanticity proves distinct features exist for the same domain at different abstraction levels. The refusal paper proves complex behaviors can be a single vector — reinforcing that persona prompts work by activating specific directional pathways.

Certainty in an Uncertain World

The trick is not eliminating uncertainty — it's containing it.

Introduce the deterministic vs non-deterministic spectrum.

The Spectrum

Non-deterministic

Skills and agents are vague by design. They reason, interpret, and adapt.

Deterministic

Tasks use exact tools, exact commands, exact checks, exact outcomes.

Even experts use tools — give your AI tools.

Emphasize that intelligent systems still need concrete interfaces to reality.

CLI Environments Are Full Service

Files. Shell. Git. Tests. Search. Network. Automation.

⚠️ Also dangerous without sandboxing, permissions, and validation.

This is why environment design matters as much as model choice.

Hooks: Determinism in the Agent Lifecycle

Automatic checks at the moments that matter.

Hooks convert good intentions into guaranteed execution points.

Hook Examples

  • Run linters on completion
  • Invoke security agents to validate code before merge
  • Chain validation: unit tests → typecheck → browser test → review
  • Reject “done” until deterministic checks pass
Hooks are one of the highest leverage investments because they scale across every run.

Most modern dev tools offer a CLI

If a human can script it, an agent can probably orchestrate it.

The CLI is the bridge between model reasoning and deterministic execution.

Why the CLI Wins

  • Full suite of arguments and flags
  • Compose with shell tools like grep, awk, jq, and pipes
  • Wrap SDKs, frameworks, package managers, and infrastructure
  • Easy to reuse inside skills, hooks, CI, and scripts
A good CLI turns agent work from brittle UI automation into robust system control.

Context Window

Your best friend and worst enemy.

Context is both power and failure mode.

Model Context Windows

Model Window Operational Reality
Claude 1M Great for codebases and long specs, but you can still drown signal in noise
GPT-4o 128K Enough for many app tasks, but requires tighter packing
Gemini 1M Excellent for long inputs, but structure still matters
A larger window helps, but does not eliminate the placement problem.

The Valley of Meh

"Lost in the Middle"

Too little context starves reasoning. Too much context dilutes relevance. The middle gets ignored.

Explain that the issue is not just volume; it is where relevant information sits inside the volume.

U-Shaped Retrieval Curve

Accuracy ^ | ████████\ /████████ | \ / | \ / | \____ Meh Valley ___/ | \ / +------------------------------------------------> Position in context Beginning End
This is the intuition pump: front and back matter more than middle.

Implications

  • Keep context small and relevant
  • Front-load long reference data, then restate instructions at the end
  • Anthropic guidance reports up to 30% improvement from long-context structure
  • Chunk, summarize, and re-anchor aggressively
The action item is structural discipline, not blind maximization of tokens.

The Goldilocks Approach

Not too little. Not too much. Just enough structure, context, and guidance.

This section turns research into prompt and workflow heuristics.

Same Task, More Tokens

Levy et al., ACL 2024 — effective reasoning often peaks at roughly 50–70% of max context.

More tokens can increase distraction, contradiction, and retrieval failures.

Bigger input is not free capacity; it can become cognitive interference.

Mimic the Training Set

  • Use Markdown for readable hierarchy
  • Use XML tags for explicit boundaries
  • Use few-shot examples for pattern completion
  • Show the shape of the output you want
Language models like familiar structure; make the task look like examples they have seen before.

Anthropic Best Practices

  1. Golden Rule: "Show your prompt to a colleague..."
  2. Long Context Structure: "Put longform data at the top... up to 30%"
  3. XML Tags: "XML tags help Claude parse complex prompts unambiguously..."
  4. Ground in Quotes: "For long document tasks, ask Claude to quote relevant parts..."
  5. Give Context/Motivation: "Providing context or motivation... helps Claude better understand..."
  6. Persona/Role: "Setting a role in the system prompt focuses Claude's behavior..."
  7. Literal Instruction Following (Opus 4.7): "interprets prompts more literally..."
  8. Effort Levels: "Start with the new xhigh effort level for coding..."
These are simple but operationally powerful defaults for prompt and workflow design.

Research Quotes That Matter

  1. Lost in the Middle (Liu et al., 2023): "Performance is highest when relevant information appears at the very beginning or very end..."
  2. Same Task More Tokens (Levy et al., 2024): "Our findings show a notable degradation..."
  3. Chain-of-Thought (Wei et al., NeurIPS 2022): "Let's think step by step" → 17.7% to 78.7%
  4. Zero-Shot Reasoners (Kojima et al., 2022): "LLMs are decent zero-shot reasoners..."
  5. Google Gemini: "We recommend to always include few-shot examples..."
The pattern across papers is remarkably consistent: structure helps, examples help, placement matters.

Using Different Models

Cognitive diversity > single-model reasoning.

Planning, implementation, review, and validation do not require the same strengths.

This is where orchestration beats hero-model thinking.

Multi-Model Debate

Diverse models debating and critiquing one another reached 91% on GSM-8K in reported research settings.

Independent failures are easier to catch than self-consistent failures.

The main idea is error decorrelation: different models miss different things.

Why Different Models Help

  • Different blind spots
  • Different training mixtures
  • Different architecture biases surface different perspectives
  • Review gets stronger when it is not performed by the same mindset that built the code
You are not only buying more intelligence; you are buying a different angle of attack.

Practical Model Allocation

Stage Model Choice Why

Building Sonnet/Opus hybrid Fast iteration + complex reasoning

The Ralph Loop: Bash-Driven Agent Cycling

Automate planning, building, checking, and retrying until the work is actually complete.

Now tie all the concepts together in one operational loop.

Core Loop Snippet


OUTPUT=$(copilot \
    --allow-all-tools \
    --model claude-opus-4.6 \
    -p "$PROMPT_CONTENT" 2>&1 | tee /dev/stderr)

git push origin "$CURRENT_BRANCH"

if echo "$OUTPUT" | grep -q "$COMPLETION_PROMISE"; then
    echo "✓ Completion promise found"
    break
fi
This pattern lets the shell own persistence while the model owns local reasoning.

Two Modes

./loop.sh -m plan

Research, decompose, assign sub-agents, create a spec.

./loop.sh -m build

Implement, validate, review, fix, and retry until exit criteria are met.

Split thinking mode from execution mode to keep each prompt sharper.

Build Prompt: Key Elements

  1. Precise task statement
  2. Repository and environment context
  3. Required validation commands
  4. Completion criteria and promise phrase
  5. Permission boundaries / tool policy
  6. Instructions for iteration after failure
The completion promise is important because it creates a machine-detectable success contract.

Elegant Insight

  • Clear context each run
  • Re-anchor every step
  • Persistent files beat fragile conversational memory
  • Bash provides the durable outer state machine
The shell is the boring hero: it gives persistence, branching, and completion semantics.

Agentic SDLC

  1. Clarify the problem
  2. Gather repository context
  3. Create a plan
  4. Review the plan
  5. Assign implementation work
  6. Write code
  7. Run deterministic validation
  8. Perform browser / system checks
  9. Review with a different agent or model
  10. Fix discovered issues
  11. Commit and ship
This is the workflow view: many agents, many checks, one delivery pipeline.

Key Principle

Use multiple agents, with deterministic validation between steps.

Do not trust a single completion message. Trust reproducible checks, layered review, and explicit exit criteria.
